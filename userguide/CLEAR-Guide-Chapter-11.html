Projects

A project is any outcome that requires more than one physical action to complete. Not a task with sub-tasks — a genuine multi-step outcome where the path isn't fully clear until you're moving through it. CLEAR keeps projects present in your system without letting them flood your daily view. Only one step surfaces at a time.

What makes something a project

The distinction between a task and a project matters. A task is a single physical action — "Call Henderson HVAC to schedule pump inspection." A project is a desired end state that requires multiple actions — "Water pump serviced before winter." The project exists to hold the outcome and the sequence of steps toward it. The task is what you actually do.

In practice, you often discover that something is a project during processing. You capture "deal with the water pump" and realize, as you write the title, that there are at least three steps: call for an estimate, approve the work, follow up on completion. That's a project. If you're not sure, ask: could I do this in one session without having to figure anything out? If the answer is no, it's probably a project.

A project is an outcome, not a category

"USCIS" is not a project — it's a hat. "I-485 application filed and accepted" is a project. The title should always be a result, stated as if it's already done. Not "Work on the I-485 filing" — "I-485 application filed and accepted." This distinction determines what done looks like, which is the most important thing a project title can tell you.

Creating a project

Projects are created in two ways: directly from the inbox when you choose the Project path during processing, or manually from the Projects tab using the + button. Either way, the same eight-step setup flow runs.

1
Outcome statement
What does done look like? Written as a result, not an activity. The system prompts: "Describe the outcome, not the work." This becomes the project title. It's worth spending a moment here — a clear outcome statement is what allows the system to suggest project linkage when you set your Big Three each week.
Example: "Henderson HVAC proposal reviewed and approved" not "Deal with HVAC proposal"
2
Hat
Which life area does this project belong to? One tap on your hat list. The hat determines the color coding throughout the project's lifetime — in the project list, on task rows, in the Weekly Architecture Session review.
3
Horizon
When do you expect to complete this? Four options — this week, this month, this quarter, or someday. Horizon determines where the project surfaces in the project list and how urgently it's treated. Projects with no realistic timeline go in Someday; they stay in the system without cluttering the active view.
Horizon can be updated at any time from the project detail view as circumstances change.
4
Deadline
Has an outside party set a real deadline for this? Yes or No. If Yes, a date picker appears. This field is strictly for hard external deadlines — a regulatory filing date, a client delivery, a publisher. Self-imposed target dates are not supported here. If no outside party has set a date, leave this blank.
Setting a deadline activates the pace status system — the progress bar that shows whether you're on track.
5
Weekly outcome
Optional. A single sentence: "What does progress look like this week?" This field resets each week and is editable from the project detail view. It gives you a weekly anchor for the project without requiring a full breakdown of every step. Useful for long-horizon projects where the weekly goal is more useful than the distant final outcome.
6
First action
The very next physical thing that needs to happen. This becomes the project's active next action — the only task from this project that will surface in the Launchpad. It should be specific and completable in one session. If you're not sure what the first step is, that's a signal: start with "Figure out what the first step is" — that's a valid task.
This is the most important step. A project without a clear next action is a project that won't move.
7
Quick Build
Brain-dump the remaining steps. One per line, as fast as you can type or speak. Don't worry about order, completeness, or phrasing — just get them out. The system accepts them as-is and queues them behind the first action. You can reorder, edit, and add more at any time from the project detail view.
Skippable. If you can only see the first step, that's enough to start.
8
People involved
Anyone whose participation the project depends on. Adds them from your contact list or creates a new contact inline. Name is the only required field at this stage. People linked to a project appear in the project detail view and get tracked in your relationship maintenance system.
Skippable. Add people later from the project detail view if you're not sure yet.

The single next action principle

A project has exactly one active next action at all times. That one task is the only step from the project that surfaces in the Launchpad. Everything else in the project is hidden from daily operations — not deleted, not forgotten, just waiting behind the current step.

The entire project list is irrelevant until the current step is done. You don't need to see the full project. You need to see the next move.

When you complete the current next action, the system prompts: "Great — [task] is done. What's the next step toward [project outcome]?" If you ran the Quick Build during setup, the next queued step is suggested. You confirm it, or choose a different one, or add a new step if the situation has changed. The project advances by one action at a time.

You can always change the designated next action from the project detail view — tap "Change next action" to select a different task from the project's queue. Use this when circumstances change and a different step has become more urgent than the current one.

Why only one? The ADHD brain is particularly susceptible to project sprawl — the experience of opening a project view, seeing fifteen steps, and becoming overwhelmed into inaction. A single visible next action eliminates that decision entirely. There is only one thing to do. That thing either fits your current context or it doesn't. The system handles the rest.

Project horizons

Horizon is how CLEAR categorizes your projects by timeframe without requiring you to manage a complex priority system. Four options:

This week
Actively in motion right now. The next action is in your daily task pool. Surfaces prominently in the Weekly Architecture Session project review.
This month
Active but not immediate. Next action is in the task pool but weighted lower than this-week projects. Still visible in the Architecture Session.
This quarter
On the horizon. Next action exists but surfaces infrequently. Reviewed in the Architecture Session to check whether horizon needs updating.
Someday
Parked. Real but not timely. Next action is inert — won't surface in the Launchpad. Appears in the Someday review, not the active project list.

Horizons are editable at any time from the project detail view. A project that was This Quarter last month might become This Week when circumstances shift. The project list defaults to showing This Week and This Month — the active layer. Tap the filter to expand to all horizons.

Deadlines and pace tracking

When a project has a hard external deadline, the project detail view shows a pace status block — an honest picture of whether you're on track to meet it.

Two bars are shown simultaneously. The solid bar is actual completion — how many tasks in the project have been done. The outline bar is expected completion at current pace — where you'd need to be to finish on time. The gap between them is the honest picture.

Three pace statuses:

On track — actual completion is within 10% of expected. Green indicator. No action prompted.

Behind — actual is more than 10% below expected. Amber indicator. The notification: "Behind pace — roughly N tasks per day to meet this deadline."

At risk — the gap is severe. Red indicator. When a deadline is within 14 days and the project is at risk, it's automatically elevated as a Big Three candidate in the next Architecture Session: "[Project] has a hard deadline in N days and is behind pace." This card cannot be dismissed without a response — you either add it to your Big Three or explicitly acknowledge the situation and move on.

At-risk projects also receive a fit score boost: all tasks within the project surface more aggressively in the Now list, giving the project active advocacy even before you consciously prioritize it.

When a project stalls

A project is flagged as stalled when its next action hasn't been started in seven or more days and no tasks in the project have been completed in that period. CLEAR surfaces a gentle weekly notification: "Your [Project Name] project hasn't had any movement in 10 days. Want to look at it?"

Stalled projects also appear prominently in the Weekly Architecture Session's project review screen — for each stalled project, the session prompts: "What is the very next physical action?" This isn't optional. The session doesn't advance until you either provide a next action, mark the project done, or park it to Someday. A stalled project without a clear next action is the most common cause of projects dying quietly without resolution.

On reactivating stalled projects

When a project stalls, the most useful question is usually the same as the one that started it: "What is the very next physical action?" Not "Why is this stalled?" or "What should I do about this project?" — just: what would I do first if I were going to work on this right now? If the answer is clear, the project isn't really stalled — it just needs a nudge. If the answer isn't clear, that's the next action: figure out what the next action is.

Adding steps over time

Projects don't need to be fully mapped at creation. The Quick Build step in setup is where you brain-dump what you can see — but for long-horizon projects or complex ones, you often can't see more than the next two or three steps from where you stand. That's fine. The system is designed for it.

From any project's detail view, the "Add steps" button opens the same Quick Build interface used during setup. Add as many steps as you can see from your current position. They queue behind the current next action and surface in sequence as earlier steps are completed.

No metadata is required when adding steps through Quick Build — you can enter titles fast and fill in energy, time estimate, and other fields from the individual task detail screen afterward. Speed matters here: the goal is to externalize the steps while you can see them, not to stop and process each one fully.
